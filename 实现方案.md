# 菜单决策 App — v1 实现方案

## 1. 项目概述

一个用于「决定吃什么」的跨平台 App，核心交互是**转盘抽签**。

- **目标平台**：iOS / Android / macOS / Windows（一份代码）
- **使用模式**：纯本地，无账号系统，无后端
- **底部导航**：转盘 / 菜单 / 推荐 / 我的

---

## 2. 技术栈

| 维度 | 选型 | 理由 |
|---|---|---|
| 框架 | **Flutter**（最新稳定版） | 一份代码全平台、自定义动画顺手、生态成熟 |
| 状态管理 | **Riverpod** | 类型安全、可派生、测试友好 |
| 本地存储 | **SharedPreferences + JSON** | v1 数据量小，避免引入代码生成器 |
| 音效 | **audioplayers** | 跨平台稳定 |
| 文件选择(头像) | **file_picker** | 一个包搞定移动+桌面 |
| 路由 | 不用 go_router | 4 个 Tab + `IndexedStack` 够用 |

### 故意不引入的依赖

drift / isar / hive / freezed / json_serializable / go_router —— 全部不在 v1 引入，避免 `build_runner` 代码生成开销，模型类手写 `toJson` / `fromJson`。

---

## 3. 关键设计决策

### 决策 1：转盘候选数 > 8 时如何展示（采用方案 B）

转盘最多 8 扇，但用户勾选参与的菜品可能更多。三种处理方式：

| 方案 | 行为 | 结论 |
|---|---|---|
| A. 静态显示 | 转盘上固定显示前 8 个，后面永远抽不到 | ❌ 不公平 |
| **B. 动态显示** ⭐ | 每次点旋转，从勾选菜品里随机抽 8 个上转盘，再从这 8 个抽中奖 | ✅ 采用 |
| C. 显示 ≠ 中奖 | 转盘显 8 个但实际从全部抽，可能停在 A 但弹窗说 X | ❌ 体验差 |

### 决策 2：转盘旋转结果由抽中决定，再控制动画指过去

不依赖物理随机（扇区夹角差异会破坏概率公平），而是**先随机选中 winner，再计算应停止的角度**。

### 决策 3：`selected`（是否参与转盘）属于 Dish 自身的持久状态

不引入 `WheelSession` 表。用户希望勾选状态跨会话保持。

### 分层原则

- `data/models`：纯数据类（`copyWith` + JSON 序列化）
- `data/repositories`：业务无关的存取
- `features/*/providers`：业务状态（Riverpod）
- `features/*/pages, widgets`：UI 层，**只读 provider，不直接碰 repository**

## 6. 存储策略

v1 **全用 `SharedPreferences` + JSON 序列化**：

| 数据 | Key | 类型 |
|---|---|---|
| 菜品列表 | `dishes_v1` | JSON 数组 |
| 用户资料 | `profile_v1` | JSON 对象 |
| 应用设置 | `settings_v1` | JSON 对象 |

**理由**：

1. 菜品规模预估 < 200 条，10KB 级别，性能完全够
2. 不用引代码生成器（drift/isar 都要 build_runner）
3. 字段迭代简单

**v2 迁移点**：加历史记录、统计后迁移到 `drift` 或 `isar`。key 带 `_v1` 后缀就是为后续 migrate 准备。

## 7. 状态管理（Riverpod）

```
dishListProvider          → List<Dish>           // 全部菜品
  └─ selectedDishesProvider → List<Dish>         // 派生：勾选的
       └─ canSpinProvider    → bool              // 派生：勾选数 >= 2

profileProvider           → UserProfile
settingsProvider          → AppSettings
  └─ appThemeProvider     → ThemeData            // 派生

wheelSessionProvider      → WheelSession?        // 当次旋转的临时状态
                                                 //   { displayedDishes, winner, isSpinning }
```

`wheelSessionProvider` 是**非持久化**的临时状态，每次点旋转重新生成。

## 8. 转盘核心算法

```text
当用户点"开始旋转":
  selected = selectedDishesProvider 当前值
  if selected.length < 2:
    提示"至少勾选 2 个"; return

  displayed = selected.length > 8
              ? shuffle(selected).take(8)
              : selected
  winnerIdx = random(0, displayed.length)
  winner    = displayed[winnerIdx]

  扇区夹角  = 360 / displayed.length
  目标角度  = -(winnerIdx * 扇区夹角 + 扇区夹角/2) + 随机抖动(<扇区夹角/2)
  最终角度  = 目标角度 + 360 * 圈数(5~8圈)

  动画(2.5~3s, easeOutQuart)
  动画结束 → 弹窗显示 winner
           → 文字提示"还有 N 个未在转盘上"(N = selected.length - displayed.length)
```

**关键点**：转盘用 `CustomPainter` + `AnimationController` 自绘自控，**不引第三方转盘包**（多数旋转结束角度不可控）。
